# Blommunity 제품 기획서

> 게시판·블로그·커뮤니티를 만들고 싶은 누구나, 코드 없이도 자기만의 커뮤니티를 열 수 있게.

---

## 1. 풀려는 문제

커뮤니티, 블로그, 카페 같은 게시판 서비스를 만들고 싶은 사람은 많습니다. 그런데 막상 시작하려면 두 갈래 길밖에 없습니다.

**1) 남의 플랫폼에 얹혀산다 (네이버 카페, 다음 카페 등)**
빠르고 쉽지만, 내 데이터가 내 것이 아니고, 디자인도 도메인도 수익화도 내 마음대로 못 합니다. 플랫폼 정책이 바뀌면 끌려다녀야 하죠.

**2) 직접 다 만든다**
자유롭지만, 회원·게시판·댓글·권한·알림·신고처럼 "어느 커뮤니티나 똑같이 필요한 것들"을 매번 처음부터 만들어야 합니다. 정작 차별화에 쓸 시간을 기본기 구현에 다 씁니다.

**Blommunity는 이 사이의 빈자리를 채웁니다.** 게시판 서비스에 공통으로 필요한 기능을 미리 다 만들어 제공하고, 사업자는 자기 색깔(디자인, 운영 정책, 수익화)에만 집중하게 합니다.

> **영감**: NHN커머스의 "샵바이"가 쇼핑몰 사업자에게 한 일 — 필요한 기능(API·디자인·CMS·SEO)을 거의 다 제공해 누구나 자기 쇼핑몰을 차리게 한 것 — 을, Blommunity는 게시판·커뮤니티 영역에서 합니다.

---

## 2. 제품 비전과 원칙

### 비전
**누구나 자기만의 커뮤니티를 빠르게 열고, 온전히 소유한다.**

### 제품 원칙
1. **공통은 우리가, 차별화는 당신이** — 반복되는 기본기는 Blommunity가 책임지고, 사업자는 정체성에 집중합니다.
2. **데이터는 사업자의 것** — 자체 도메인, 데이터 소유권, 운영 자율성을 보장합니다.
3. **열려 있다 (오픈소스)** — 코드가 공개되어 신뢰할 수 있고, 직접 호스팅하거나 확장할 수 있습니다.
4. **선택할 수 있다** — 코드 없이 기본 템플릿으로 바로 쓰거나, API만 가져다 자기 프론트를 붙이거나, 둘 다 가능합니다.

## 3. 누구를 위한 제품인가

Blommunity에는 세 종류의 사용자가 있습니다. 각자의 입장과 "이걸로 무엇이 가능해지는가"가 다릅니다.

### 3.1 운영자 (Admin) — Blommunity를 운영하는 우리
**누구인가**: 플랫폼 자체를 관리하는 내부 인력.
**무엇을 하는가**: 어떤 사업자가 들어왔는지 관리하고, 가입을 승인하고, 플랫폼이 건강하게 돌아가는지 살핍니다.
**무엇이 가능해지는가**: 신뢰할 수 있는 사업자만 받아들이고, 문제가 생기면 추적할 수 있습니다(활동 기록).

### 3.2 테넌트 (Tenant) — 게시판 사업을 하려는 사업자 (핵심 고객)
**누구인가**: 커뮤니티·블로그·카페를 운영하고 싶은 사업자. 동호회 운영자, 1인 크리에이터, 소규모 미디어, 자기 서비스에 커뮤니티를 붙이려는 스타트업 등.
**무엇을 하는가**: 가입 후 자기 공간을 만들고, 게시판을 구성하고, 회원과 권한을 관리하고, 디자인을 고르고, 운영(신고 처리·통계)합니다.
**무엇이 가능해지는가**: 개발 없이(또는 최소한의 개발로) 자기 커뮤니티를 빠르게 열고, 운영에만 집중할 수 있습니다.

### 3.3 사용자 (User) — 커뮤니티를 실제로 쓰는 사람
**누구인가**: 테넌트가 운영하는 커뮤니티의 최종 이용자.
**무엇을 하는가**: 가입하고, 글과 댓글을 쓰고, 반응하고, 공간/게시판에 참여합니다.
**무엇이 가능해지는가**: 익숙한 게시판 경험(공개/비공개, 멤버십, 대댓글 등)을 그대로 누립니다.

## 4. 제품의 구조 (3개의 면)

Blommunity는 세 사용자에 대응하는 세 개의 "면"을 가집니다.

| 면 | 사용자 | 한 줄 설명 | 제공 형태 |
|----|--------|------------|-----------|
| **운영자 어드민** | 운영자 | 플랫폼을 운영하는 관제탑 | 백엔드 + 프론트엔드 |
| **테넌트 콘솔 (CMS)** | 테넌트 | 사업자가 자기 커뮤니티를 만들고 운영하는 작업실 | 백엔드 + 프론트엔드 |
| **커뮤니티** | 최종 이용자 | 실제 게시판이 돌아가는 곳 | 백엔드 API + 기본 템플릿 (또는 직접 구축) |

> **핵심 제공 전략**: 커뮤니티 영역은 "백엔드까지는 우리가 완성해서 주고, 화면은 기본 템플릿을 제공하되 직접 만들 수도 있게" 합니다. 비개발자는 템플릿으로 바로 시작하고, 개발자는 API로 자유롭게 구축하는 — 양쪽 모두를 끌어안는 구조입니다.

## 5. 핵심 개념 (제품의 세계관)

Blommunity 안의 공간은 다음처럼 층을 이룹니다.

```
테넌트(Tenant)  ─ 사업자 하나
  └─ 공간(Space)  ─ 운영하는 사이트/커뮤니티 단위 (여러 개 가능)
       └─ 게시판(Board)  ─ 공간 안의 주제별 게시판
            └─ 게시물(Post)
                 └─ 댓글(Comment)  ─ 대댓글 지원
```

- **공간(Space)을 따로 둔 이유**: 한 사업자가 여러 커뮤니티를 운영할 수 있게 하기 위해서입니다. 예를 들어 한 회사가 "고객 커뮤니티"와 "직원 라운지"를 각각 운영하는 식입니다.
- **멤버십과 역할**: 사용자는 공간/게시판에 각각 멤버로 참여하며, 소유자·관리자·일반회원 같은 역할을 가집니다. 초대받아 대기 중인 상태(PENDING)와 실제 멤버(ACTIVE)를 구분합니다.
- **공개 범위**: 모든 공간·게시판·게시물은 공개(누구나)와 비공개(멤버만)를 선택할 수 있습니다.
- **되돌릴 수 있는 삭제**: 삭제는 즉시 사라지지 않고 복구 가능한 형태로 처리됩니다(운영 안정성).

## 6. 핵심 사용 시나리오

기능을 나열하기 전에, "사람들이 실제로 이걸로 무엇을 하는가"를 그려봅니다.

### 시나리오 A — 사업자가 커뮤니티를 연다
> 독서 모임을 키우고 싶은 지은 씨는 Blommunity에 가입한다. 운영자의 승인을 받은 뒤, "북클럽"이라는 공간을 만들고 그 안에 '자유게시판'과 '이달의 책' 게시판을 만든다. 기본 템플릿을 골라 로고와 색을 바꾸고, 자기 도메인을 연결한다. 30분 만에 커뮤니티가 열린다.

### 시나리오 B — 멤버가 참여한다
> 회원 민수 씨는 북클럽 커뮤니티에 가입한다. '자유게시판'은 공개라 바로 글을 읽지만, '운영진 전용' 게시판은 비공개라 초대를 받아야 들어갈 수 있다. 글을 쓰고, 좋아요를 누르고, 대댓글로 대화한다.

### 시나리오 C — 사업자가 운영한다
> 지은 씨는 콘솔에서 신고된 글을 확인하고 처리한다. 누가 새로 가입했는지, 어떤 글이 인기인지 통계로 본다. 말썽을 부리는 회원에게는 권한을 조정하거나 차단한다.

### 시나리오 D — 개발자가 자기 식으로 붙인다
> 스타트업 개발자 현우 씨는 기본 템플릿 대신 API만 가져다 자사 앱에 어울리는 커뮤니티 화면을 직접 만든다. 회원·게시판·댓글 로직은 Blommunity가 처리하므로, 그는 화면과 자사 서비스 연동에만 집중한다.

## 8. 차별화 포인트

| 차별점 | 의미 | 누구에게 와닿는가 |
|--------|------|-------------------|
| **공통 기능 완비** | 게시판에 필요한 기본기를 다 제공 | 빨리 시작하고 싶은 사업자 |
| **데이터 소유 + 자체 도메인** | 내 커뮤니티를 온전히 소유 | 플랫폼 종속이 싫은 운영자 |
| **오픈소스** | 신뢰·확장·직접 호스팅 가능 | 개발자, 기술에 민감한 고객 |
| **헤드리스 + 기본 템플릿** | 코드 없이도, 코드로도 가능 | 비개발자 + 개발자 모두 |
| **여러 공간 운영** | 한 사업자가 여러 커뮤니티 운영 | 규모 있는 사업자 |
| **한국 커뮤니티 친화** | 익숙한 카페형 경험, 카카오 로그인 등(예정) | 국내 사업자 |

## 9. 무엇이 되어 있고, 무엇이 남았나

### 이미 되어 있는 것 (제품의 뼈대)
커뮤니티가 돌아가기 위한 **핵심 골격**이 완성되어 있습니다. 사업자가 가입해 공간과 게시판을 만들고, 회원이 참여해 글과 댓글을 주고받는 흐름, 그리고 이 모든 것을 떠받치는 회원·권한·멤버십·승인 체계가 동작합니다.

### 다음에 채울 것 (실사용을 위한 살)
지금 구조에서 "실제로 쓸 만한 커뮤니티"가 되려면, 사람들이 당연하게 기대하는 경험이 더 필요합니다.
- **이미지·파일 첨부** — 사진 못 올리는 게시판은 사실상 못 씁니다. (최우선)
- **좋아요/반응** — 커뮤니티의 가장 기본적인 상호작용.
- **태그/분류** — 글이 쌓이면 분류가 필요합니다.
- 이후 **알림 → 신고/모더레이션 → 검색 → 소셜 로그인** 순으로 확장.

### 그 다음 (제품의 얼굴)
- **테넌트 콘솔 화면** — 사업자가 실제로 만지는 작업실.
- **커뮤니티 기본 템플릿** — 비개발자가 바로 쓸 수 있는 화면.
- **운영자 어드민 화면**.

## 10. 로드맵 (제품 단계별 목표)

각 단계는 "이 단계가 끝나면 사용자에게 무엇이 가능해지는가"로 정의합니다.

### 1단계 — "쓸 만한 게시판" 완성
> **사업자가 만든 커뮤니티에서, 회원이 사진 올리고 좋아요 누르며 진짜로 활동할 수 있다.**
- 이미지·파일 첨부, 좋아요/반응, 태그
- 현재 골격의 완성도·안정성 다지기

### 2단계 — "운영되는 게시판"
> **사업자가 커뮤니티를 건강하게 관리할 수 있다.**
- 알림, 신고·모더레이션, 검색, 조회수

### 3단계 — "코드 없이 운영하는 게시판"
> **비개발 사업자도 화면을 보며 직접 커뮤니티를 운영한다.**
- 테넌트 콘솔(CMS) 화면

### 4단계 — "관제되는 플랫폼"
> **운영자가 전체 플랫폼을 한눈에 관리한다.**
- 운영자 어드민 화면, 플랫폼 통계

### 5단계 — "바로 쓰는 게시판"
> **가입하면 곧장 보이는 커뮤니티 화면이 있다.**
- 커뮤니티 기본 템플릿(블로그형/카페형), 자체 도메인 연결
