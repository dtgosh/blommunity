# Blommunity 로드맵

> 각 버전을 **"이 버전이 끝나면 무엇이 가능해지는가"**로 정의합니다. 버전 번호는 약속의 단위입니다.
>
> 이 문서는 [feature-spec.md](feature-spec.md)의 마일스톤 M1~M6을 v1.0 출시 관점으로 다시 묶은 것입니다. 둘의 매핑은 4절 표에 있습니다.

---

## 1. v1.0의 정의 — "헤드리스 백엔드 1.0"

v1.0은 **"API가 굳었고, 신뢰하고 의존해도 된다"**는 약속입니다.

- **무엇이 약속되는가**: 핵심 도메인(테넌트·공간·게시판·게시물·댓글), 인증·권한, 미디어, 상호작용이 API로 안정적으로 제공되고, `/v1/` 표면이 깨지지 않습니다([api-versioning-policy.md](api-versioning-policy.md)).
- **무엇이 미뤄지는가**: **UI(테넌트 콘솔·운영자 어드민)는 v2.0 이후**입니다. v1.0은 헤드리스 백엔드이며, 화면은 기본 템플릿/사업자 자체 프론트로 붙입니다.
- **왜 이렇게 긋는가**: "API를 믿어도 된다"를 먼저 확실히 한 뒤에 화면을 얹는 것이, 사업자가 자기 프론트를 올리는 헤드리스 제품에 맞는 순서이기 때문입니다.

---

## 2. v1.0까지의 버전 로드맵

현재 위치: **v0.1.x** (핵심 도메인·인증·권한 골격 완료).

### v0.2.0 — Hardened (격리·인증 견고화)
> **끝나면 가능해지는 것: 테넌트 데이터가 코드 실수로도 새지 않고, 토큰이 탈취돼도 피해 창이 좁다.**

- `X-SC-07` 테넌트 격리 자동화 ⭐ — Prisma Client Extension([ADR 0001](adr/0001-tenant-isolation.md))
- `X-SC-08` RefreshToken 회전 + `T-AC-06` 토큰 갱신 — AT(15분)+RT(14일)([ADR 0003](adr/0003-refresh-token.md))
- `X-SC-09` 비밀번호 정책 강화
- `X-SC-10` 로그인 시도 제한(rate limit)

→ Blommunity의 가치: 멀티테넌트 SaaS의 **신뢰 기반**을 먼저 다집니다. 이게 안 되면 그 위에 무엇을 얹어도 불안합니다.

### v0.3.0 — Media (파일·이미지)
> **끝나면 가능해지는 것: 게시물에 사진과 파일을 올릴 수 있다.**

- `X-MD-01` 파일 업로드 + `X-MD-02` S3 호환 스토리지 연동([ADR 0002](adr/0002-storage-adapter.md))
- `U-PT-06` 게시물 이미지/파일 첨부
- `X-MD-03` 이미지 리사이즈(구현 위치는 [ADR 0002](adr/0002-storage-adapter.md)에서 열어 둠)
- `X-MD-05` 테넌트별 스토리지 사용량 추적

→ Blommunity의 가치: "사진 못 올리는 게시판은 사실상 못 쓴다"는 가장 큰 막힘을 풉니다.

### v0.4.0 — Interactions (좋아요·태그·멤버 관리)
> **끝나면 가능해지는 것: 회원이 반응하고 분류하며, 멤버를 관리할 수 있다.**

- `U-PT-07` / `U-CM-07` 게시물·댓글 좋아요/반응
- `U-PT-08` 게시물 태그
- `U-PT-10` 게시물 조회수
- `T-SP-13` / `T-SP-14` / `T-SP-15` 공간 멤버 목록·역할 변경·탈퇴
- `T-BD-13` / `T-BD-14` / `T-BD-15` 게시판 멤버 목록·역할 변경·탈퇴

→ Blommunity의 가치: 커뮤니티가 "글만 쌓이는 곳"에서 "상호작용하는 곳"이 됩니다.

### v0.5.0 — Stable (배치·모니터링·문서 정비)
> **끝나면 가능해지는 것: 데이터 정합성이 자동으로 유지되고, 운영 상태를 관측할 수 있다.**

- `X-DT-04` 카운터 정합성 재계산(배치) · `X-DT-05` 영구 삭제(배치) · `X-DT-06` 만료 PENDING 초대 정리(배치)
- `X-OP-07` 메트릭/모니터링 · `X-OP-08` 에러 추적
- 문서 정비(이 문서들 포함), [nfr.md](nfr.md)의 초기 목표값을 실측으로 교정

> **배치 앱 관련 메모**: 배치 작업(X-DT-04/05/06)은 스케줄러 독립 실행을 전제로 합니다. 의존성(`@nestjs/schedule`)과 분리 실행 흐름(`start:dev batch`)은 준비되어 있고, 실제 배치 잡 구현이 이 단계에서 들어옵니다.

→ Blommunity의 가치: "굳었다"를 증명하는 안정성 계층. v1.0 직전 마지막 다지기.

### v1.0.0 — 태깅
> **끝나면 가능해지는 것: 누구나 이 API에 의존해 자기 서비스를 올릴 수 있다.**

- **신규 작업 없음.** v0.5.0 끝에서 표면을 동결하고 태깅합니다.
- 이 순간부터 `/v1/`은 안정 표면이 됩니다 — 깨지는 변경 금지([api-versioning-policy.md](api-versioning-policy.md)).

→ Blommunity의 가치: "헤드리스 백엔드 1.0" 약속의 발효일.

## 3. v1.0 이후

### v1.1.0 — 신고·알림
> **끝나면 가능해지는 것: 부적절한 콘텐츠를 신고·처리하고, 회원이 알림을 받는다.**

- `U-PT-11` 게시물 신고 · `U-CM-09` 댓글 신고
- `T-OP-01` 신고 접수 조회 · `T-OP-02` 신고 처리
- `U-NT-01` 인앱 알림 목록 · `U-NT-02` 알림 읽음 처리

→ 모더레이션과 재방문 유도(알림)는 "건강하게 운영되는 커뮤니티"의 핵심이지만, v1.0의 안정 표면을 먼저 굳힌 뒤 얹습니다.

### v2.0+ — UI 계층 (먼 단계)
헤드리스 백엔드 위에 화면을 얹는 단계입니다. [sitemap.md](sitemap.md)가 그 출발점입니다.

- 테넌트 콘솔(CMS) 화면 — feature-spec 5.2 / 원래 마일스톤 **M3**
- 운영자 어드민 화면 — feature-spec 5.1 / 원래 마일스톤 **M4**
- 커뮤니티 기본 템플릿 — feature-spec 5.3 / 원래 마일스톤 **M5**
- 소셜 로그인(`U-AC-07`), 이메일 인증·재설정(`U-AC-08`/`09`), 사이트 설정(`T-ST-*`), 검색(`U-PT-12`, `X-SR-*`), 금칙어(`T-OP-03`) 등 — 시점은 이후 단계에서 확정

## 4. feature-spec 마일스톤(M1~M6) ↔ 버전 매핑

feature-spec.md의 M1~M6은 "능력 단위" 묶음이고, 위 v0.2~v1.0은 "v1.0 출시를 향한 백엔드 시퀀스"입니다. 둘은 다음처럼 대응합니다(다대다).

| 마일스톤 (feature-spec §7) | 핵심 내용 | 대응 버전 |
|---------------------------|-----------|-----------|
| **M1** 게시판이 진짜로 굴러간다 | X-SC-07, T-AC-06 | → **v0.2.0** |
| | X-MD-01·02, U-PT-06 | → **v0.3.0** |
| | U-PT-07/U-CM-07, U-PT-08 | → **v0.4.0** |
| **M2** 운영할 수 있게 된다 | U-PT-10(조회수) | → **v0.4.0** |
| | X-DT-04·05·06(배치) | → **v0.5.0** |
| | U-NT-01·02, U-PT-11/U-CM-09, T-OP-01·02 | → **v1.1.0** |
| | U-PT-12 검색, T-OP-03 금칙어 | → v1.1+ (시점 미정) |
| **M3** 사업자가 직접 운영한다 | 테넌트 콘솔 화면(5.2) | → **v2.0+** |
| **M4** 운영자가 전체를 관리한다 | 운영자 어드민 화면(5.1) | → **v2.0+** |
| **M5** 회원이 바로 쓴다 | 커뮤니티 템플릿(5.3), T-ST-02·03 | → **v2.0+** |
| **M6** 오픈소스 정식 공개 | U-AC-07·08·09, T-ST-* | → **v2.0+** (시점 미정) |

> 읽는 법: **M1·M2의 백엔드 부분이 v0.2~v0.5와 v1.1로 흩어지고, UI 중심인 M3·M4·M5와 M6은 v2.0 이후로 미뤄집니다.** 이것이 "v1.0 = 헤드리스 백엔드"라는 정의의 구체적 결과입니다.
