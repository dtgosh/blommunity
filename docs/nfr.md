# Blommunity 비기능 요구사항 (NFR)

> 기능이 "무엇을 하는가"라면, 비기능 요구사항은 **"얼마나 잘 하는가"**입니다.
>
> v1.0의 "API가 굳었다"는 약속은 측정 가능한 목표로 뒷받침되어야 합니다. 이 문서는 그 목표를 숫자로 적습니다. 단, 여기 숫자는 **초기 목표**이며 v1.0 안정화 과정에서 실측을 보고 재조정합니다. "지킬 수 있는 약속"을 점차 좁혀가는 것이 목적입니다.

---

## 1. 성능 (응답 시간)

API 응답 시간 목표입니다. 정상 부하에서의 서버 처리 시간(네트워크 왕복 제외) 기준입니다.

| 작업 유형 | P50 | P95 | P99 |
|-----------|-----|-----|-----|
| 단순 조회 (GET, 단건/소량 목록) | < 100ms | **< 300ms** | < 600ms |
| 쓰기 (POST/PATCH/DELETE) | < 150ms | **< 500ms** | < 1000ms |
| 복합 조회 (검색·집계) | < 300ms | < 800ms | < 1500ms |

- **P95가 1차 약속선**입니다. P99는 꼬리 지연(tail latency) 감시용입니다.
- 측정 위치: API 서버 내부 처리 시간. CDN·정적 파일·외부 스토리지 전송은 별도 지표로 봅니다.
- **이 값은 초기 목표입니다.** v1.0 안정화 중 실제 트래픽으로 재조정합니다.

> 성능을 떠받치는 구조는 이미 있습니다: UUID v7(시간 정렬), 핵심 인덱스(`Post[boardId, createdAt desc]`, `Comment[postId, parentId, createdAt desc]` 등), 카운터 비정규화(`commentCount`/`replyCount`), Valkey 캐시(X-OP-05). 자세한 인덱스는 [erd.md](erd.md)를 참고하세요.

## 2. 가용성

- **명시적 SLA(예: 99.9%)는 v1.0에서 약속하지 않습니다.** 운영 규모·체계가 아직 그 약속을 보증할 단계가 아니기 때문입니다. 과장된 SLA보다 정직한 목표가 신뢰에 낫습니다.
- 대신 **정성적 설계 목표**를 둡니다.
  - **단일 장애 지점(SPOF)이 없도록 설계한다.** API 서버는 무상태(stateless)로 두어 수평 확장·교체가 가능하게 합니다. 상태는 DB(PostgreSQL)와 캐시(Valkey)에 둡니다.
  - **헬스체크 엔드포인트**(`/health`, X-OP-01, ✅)로 상태를 외부에서 관측 가능하게 합니다.
  - **graceful shutdown** — 배포·재시작 시 처리 중인 요청을 끊지 않고 마무리합니다.
  - 모니터링·에러 추적(X-OP-07/08, 📅)이 갖춰지면 가용성 목표를 수치화하는 것을 검토합니다.

---

## 3. 확장성 (용량 상한)

한 테넌트가 감당 가능한 합리적 상한입니다. v1.0 단일 DB + `tenant_id` 모델 기준의 설계 가정이며, 하드 리밋이 아니라 "이 규모까지는 정상 동작을 목표로 한다"는 선입니다.

| 대상 | 테넌트당 목표 상한 |
|------|---------------------|
| 회원(User) | 10만 |
| 공간(Space) | 1,000 |
| 게시판(Board) | 공간당 100 |
| 게시물(Post) | 100만 |
| 댓글(Comment) | 게시물당 1만 |

- 전체 시스템(모든 테넌트 합산)은 위 값 × 테넌트 수까지 단일 DB로 감당하는 것을 목표로 하되, 한계가 보이면 읽기 복제본·파티셔닝·테넌트 샤딩을 검토합니다(v1.0 범위 밖).
- 대용량 목록은 항상 페이지네이션으로 끊어 읽습니다(4절).

---

## 4. Rate Limit (호출 제한)

무차별 대입·폭주를 막는 초기 정책안입니다(X-SC-10, 📅). 식별 기준은 인증 사용자 + IP 조합을 기본으로 합니다.

| 대상 | 제한 | 이유 |
|------|------|------|
| 로그인 시도 | **분당 5회** | 무차별 대입 방지 |
| 회원가입 | 분당 3회 | 스팸 계정 방지 |
| 일반 API 호출 | **분당 60회** | 폭주 방어 |
| 쓰기 작업(작성/수정) | 분당 30회 | 도배 방지 |
| 파일 업로드 URL 발급 | 분당 20회 | 스토리지 남용 방지 |

- 초과 시 **HTTP 429 (Too Many Requests)**와 `Retry-After` 헤더로 응답합니다.
- 이 수치는 초기값이며, 실사용 패턴을 보고 테넌트 등급별로 조정할 수 있습니다.

---

## 5. 페이지네이션

목록 조회는 **항상** 페이지네이션을 거칩니다. 클라이언트가 한도를 지정하지 않아도 서버가 상한을 강제합니다(전체 조회 자체를 차단).

| 항목 | 값 |
|------|-----|
| 기본 page size | **20** |
| 최대 page size | **100** |
| 정렬 기본값 | `createdAt desc` (UUID v7로 안정적) |

- `limit`이 최대값을 넘으면 최대값으로 **잘라서** 처리합니다(에러 대신 클램프).
- 커서 기반(cursor) 페이지네이션을 기본으로 지향합니다 — UUID v7이 시간 정렬되므로 커서로 적합하고, 깊은 offset의 성능 저하를 피할 수 있습니다.

## 6. 요청·파일 크기 제한

| 항목 | 제한 | 비고 |
|------|------|------|
| 요청 본문(JSON) | **1MB** | 일반 API. 초과 시 413 |
| 첨부 파일 (단일) | **100MB** | presigned URL 업로드 경로([ADR 0002](adr/0002-storage-adapter.md)) |
| 게시물 본문 길이 | 별도 상한 검토 | 매우 긴 본문은 별도 정책 |
| 테넌트 스토리지 총량 | 등급별 쿼터 | 사용량 추적 X-MD-05(📅) |

- 파일은 JSON 본문에 싣지 않고 **스토리지로 직접 업로드**하므로, 1MB body 제한과 100MB 파일 제한은 별개입니다.

## 7. 보안 목표

수치로 표현 가능한 보안 기준입니다. 배경과 전략은 [threat-model.md](threat-model.md), [ADR 0001](adr/0001-tenant-isolation.md), [ADR 0003](adr/0003-refresh-token.md)에 있습니다.

### 비밀번호 정책
- **현재: 최소 8자**(X-SC-09, 🛠 일부 구현).
- **강화 목표**: 최소 길이 상향 + 복잡도/유출 비밀번호 차단 검토. bcrypt 해싱은 적용됨(X-SC-04, ✅, `password VarChar(60)`).

### 토큰 수명
- **Access Token: 15분**, **Refresh Token: 14일(회전)** — [ADR 0003](adr/0003-refresh-token.md)에서 결정.
- `JWT_SECRET`은 최소 32자(현재 Joi 검증으로 강제, X-OP-04 ✅).

### 세션 정책
- 헤드리스 Bearer 토큰 기반(쿠키/서버 세션 아님).
- RT 회전으로 토큰 재사용을 감지하고, 비정상 시 세션 패밀리를 무효화합니다.
- 비밀번호 변경 시 기존 RT를 무효화하는 것을 권장(구현 시 확정).

### 전송·격리
- 운영 환경은 **HTTPS 강제**(TLS 종단은 인프라/리버스 프록시 책임).
- **테넌트 격리는 코드 누락이 불가능한 구조로 강제**(X-SC-07, [ADR 0001](adr/0001-tenant-isolation.md)) — Blommunity에서 가장 중요한 보안 목표.

---

## 8. 관측 가능성 (Observability)

목표를 "측정 가능"하게 하려면 관측이 전제입니다.

- **구조화 로깅(Winston, JSON)** — 운영 환경(X-OP-03, ✅).
- **메트릭/모니터링** — OpenTelemetry·Prometheus 연동(X-OP-07, 📅). 위 P50/P95/P99를 실제로 측정하는 토대.
- **에러 추적** — Sentry 등(X-OP-08, 📅).
- 이들이 v0.5.0 Stable에서 갖춰지면, 이 문서의 초기 목표값을 실측으로 교정합니다.
