# Blommunity 위협 모델 (STRIDE)

> 멀티테넌트 SaaS 백엔드에서 무엇이 잘못될 수 있는지를 STRIDE 분류로 정리합니다.
>
> 이 문서의 목적은 "완벽하다"가 아니라 **"어디가 약한지 솔직하게 안다"**입니다. 각 위협마다 현재 대응 상태를 ✅(구현됨)/🛠(진행·일부)/📅(예정)로 표시하고, 관련 기능 ID를 함께 답니다.

STRIDE는 여섯 가지 위협 범주의 머리글자입니다.

| 글자 | 위협 | 한 줄 |
|------|------|-------|
| **S** | Spoofing (위장) | 남인 척 한다 |
| **T** | Tampering (변조) | 데이터를 함부로 바꾼다 |
| **R** | Repudiation (부인) | 한 일을 안 했다고 잡아뗀다 |
| **I** | Information Disclosure (정보 노출) | 보면 안 될 걸 본다 |
| **D** | Denial of Service (서비스 거부) | 못 쓰게 만든다 |
| **E** | Elevation of Privilege (권한 상승) | 권한 밖의 일을 한다 |

> **이 중 Blommunity에 가장 치명적인 것은 I(정보 노출) — 구체적으로 테넌트 간 데이터 누출입니다.** 멀티테넌트 제품에서 옆 테넌트의 데이터가 새는 것은 단일 사고로 신뢰가 무너지는 사건입니다. 그래서 여기에 ⭐를 답니다.

---

## S — Spoofing (위장)

남의 신원을 가로채 그 사람인 척 행동하는 위협.

| 위협 | 현재 대응 | 상태 | 관련 ID |
|------|-----------|:----:|---------|
| 비밀번호 추측·탈취로 로그인 | bcrypt 해싱(salt 자동), 비밀번호 정책(현재 8자 이상 → 강화 예정) | 🛠 | X-SC-04, X-SC-09 |
| JWT 위조 | `JWT_SECRET`(최소 32자)로 서명·검증, 서명 불일치 시 거부 | ✅ | X-SC-01 |
| **JWT 탈취 후 재사용** | 현재 access token만 발급되고 만료가 짧지 않아 탈취 시 장기 악용 가능 → **RefreshToken 회전**으로 access token을 단명(15분)화 | 📅 | X-SC-08, T-AC-06 |
| 무차별 대입 로그인 | 로그인 시도 횟수 제한(분당 5회) | 📅 | X-SC-10 |

> 가장 큰 보강 포인트는 **토큰 수명**입니다. 지금은 만료가 명시되지 않은 access token 하나만 발급합니다. AT(15분) + RT(14일, 회전) 구조로 가면 탈취된 토큰의 유효 창이 좁아집니다. 자세한 결정은 [ADR 0003](adr/0003-refresh-token.md)을 참고하세요.

---

## T — Tampering (변조)

요청·데이터를 허가 없이 변조하는 위협.

| 위협 | 현재 대응 | 상태 | 관련 ID |
|------|-----------|:----:|---------|
| 잘못된/악의적 입력 주입 | `class-validator` + `ValidationPipe`로 DTO 검증, Joi로 환경변수 검증 | ✅ | X-SC-05, X-OP-04 |
| 응답 헤더·XSS·클릭재킹 등 | Helmet 보안 헤더 | ✅ | X-SC-06 |
| 과대 페이로드로 자원 변조·소모 | 요청 body 크기 제한(예: 1MB), 첨부 파일 크기 제한 | 📅 | (NFR) |
| SQL/쿼리 변조 | Prisma 파라미터 바인딩(ORM)로 기본 방어. raw query 사용 시 별도 주의 | ✅ | — |

> raw query는 ORM의 자동 방어 밖에 있습니다. [ADR 0001](adr/0001-tenant-isolation.md)에서 raw query가 테넌트 격리·변조 양쪽에서 별도 처리 대상임을 명시합니다.

---

## R — Repudiation (부인)

"나는 그런 적 없다"고 부인하는 것을 막는 위협 범주.

| 위협 | 현재 대응 | 상태 | 관련 ID |
|------|-----------|:----:|---------|
| 운영자가 한 행위를 부인 | **AdminAuditLog** — 모든 운영자 요청(method/path/query/body)을 자동 기록 | ✅ | A-AL-01 |
| 감사 로그에 민감 정보가 남음 | password 등 민감 필드 자동 마스킹 | ✅ | A-AL-03, X-SC-11 |
| 로그 위·변조 | `AdminAuditLog`는 추가 전용(`updatedAt`/`deletedAt` 없음) | ✅ | — |

> 운영자 활동 추적은 이미 구현되어 있습니다(페르소나 **하루**가 가장 두려워하는 "사고가 났는데 누가 했는지 모르는 일"에 대한 답). 다만 **테넌트 사용자 행위에 대한 감사 로그는 아직 없습니다** — 신고·모더레이션(M2) 단계에서 필요성을 재검토합니다.

---

## I — Information Disclosure (정보 노출) ⭐

**Blommunity에서 가장 중요한 위협 범주.**

| 위협 | 현재 대응 | 상태 | 관련 ID |
|------|-----------|:----:|---------|
| **⭐ 테넌트 간 데이터 누출** | 테넌트 스코프 모델에 `tenant_id`. 단, 현재는 쿼리마다 `{ tenantId }`를 **수동 전달** → 한 곳이라도 누락하면 격리가 깨짐 | 🛠 | **X-SC-07** |
| 비공개 공간/게시판 내용 노출 | Visibility(PUBLIC/PRIVATE) + 멤버십 검증 | ✅ | X-SC-03 |
| 응답에 비밀번호 등 민감 필드 포함 | Entity 직렬화 시 민감 필드 제외, 감사 로그 마스킹 | ✅ | X-SC-11 |
| 다른 회원의 비공개 정보 조회 | 같은 테넌트 내 조회로 제한, User `visibility` 기본 PRIVATE | ✅ | U-AC-06 |

> **왜 ⭐인가.** 테넌트 격리는 "모든 쿼리가 빠짐없이 `tenant_id` 필터를 가져야 한다"는 전수(全數) 조건입니다. 99개 쿼리가 맞아도 1개가 틀리면 그 1개로 옆 테넌트 데이터가 샙니다. 현재 코드는 컨트롤러에서 `findOne({ id, tenantId: user.tenantId, deletedAt: null })`처럼 손으로 전달하므로, 휴먼 에러 한 번이 전체 신뢰를 깹니다.
>
> **대응**: Prisma Client Extension으로 `tenant_id` 주입을 자동·강제화합니다. 이것이 v0.2.0 Hardened의 최우선 과제이며, 결정의 전모는 [ADR 0001](adr/0001-tenant-isolation.md)에 있습니다.

---

## D — Denial of Service (서비스 거부)

자원을 고갈시켜 정상 사용을 막는 위협.

| 위협 | 현재 대응 | 상태 | 관련 ID |
|------|-----------|:----:|---------|
| API 폭주 호출 | Rate limit(일반 API 분당 60회, 로그인 분당 5회) | 📅 | X-SC-10 |
| 거대한 목록 조회로 DB·메모리 고갈 | 페이지네이션 강제(기본/최대 page size 고정) | 📅 | (NFR) |
| 대용량 파일 업로드 | 첨부 파일 크기 제한, 테넌트별 스토리지 사용량 추적 | 📅 | X-MD-05 |
| 과대 요청 본문 | body 크기 제한(예: 1MB) | 📅 | (NFR) |

> DoS 대응은 대부분 📅입니다. 구체적 수치(rate limit, page size, body·파일 크기)는 [nfr.md](nfr.md)에서 초기 정책으로 못 박습니다. 페이지네이션 **강제**(클라이언트가 limit를 안 줘도 서버가 상한을 건다)가 특히 중요합니다.

---

## E — Elevation of Privilege (권한 상승)

자기 권한을 넘는 일을 수행하는 위협.

| 위협 | 현재 대응 | 상태 | 관련 ID |
|------|-----------|:----:|---------|
| 사용자 토큰으로 운영자 API 호출 | **TokenType 분리** — `USER`/`ADMIN` 토큰을 가드에서 교차 차단(`@AdminOnly()`) | ✅ | X-SC-02 |
| 낮은 역할이 높은 권한 행위 수행 | 역할 계층 인가(`OWNER > MANAGER > MEMBER`), `@Role()` 가드 | ✅ | X-SC-03 |
| 본인 권한·승인 자가 조작 | 본인 권한 변경·본인 승인 회수·본인 삭제 금지 규칙 | ✅ | A-AM-05·06·07, T-UM-03·04 |
| 다른 테넌트의 권한으로 행위 | 사용자 토큰에 `tenantId` 포함, 운영자 토큰에는 미포함(전역) | ✅ | X-SC-02, X-SC-07 |

> 권한 모델은 코드에 비교적 단단히 박혀 있습니다. 가드는 토큰 타입(`USER` vs `ADMIN`)과 역할 레벨을 둘 다 검사하고, "본인에 대한 위험 행위"를 명시적으로 금지합니다. 권한 매트릭스 전체는 [feature-spec.md](feature-spec.md) 1.2를 참고하세요.

---

## 요약 — 지금 가장 시급한 것

위협을 한 줄로 줄이면 우선순위가 보입니다.

1. **테넌트 격리 자동화 (X-SC-07, I 범주 ⭐)** — 수동 격리의 누락 위험을 구조로 제거. → [ADR 0001](adr/0001-tenant-isolation.md)
2. **토큰 수명 단축 + RefreshToken 회전 (X-SC-08, S 범주)** — 탈취 토큰의 유효 창 축소. → [ADR 0003](adr/0003-refresh-token.md)
3. **Rate limit + 페이지네이션 강제 (X-SC-10, D 범주)** — 폭주·고갈 방어. → [nfr.md](nfr.md)

이 셋이 v0.2.0 Hardened의 골자입니다. 나머지(감사 로그 확장, 입력 검증 강화 등)는 그 위에 얹습니다.
