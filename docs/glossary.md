# Blommunity 용어집

> 같은 단어를 다른 뜻으로 쓰면 문서도 코드도 흔들립니다. 이 문서는 Blommunity에서 쓰는 핵심 용어의 **정의**와 **사용 규칙**을 고정합니다.
>
> 다른 문서를 쓸 때 여기 정의를 따릅니다. 새 용어가 생기면 여기에 먼저 추가합니다.

---

## 1. 세 주체

Blommunity에는 세 종류의 사람이 있습니다. 이들을 혼동하지 않는 것이 가장 중요합니다.

### Admin / 운영자
Blommunity **자체**를 운영하는 내부 인력입니다. 어떤 사업자를 받을지 승인하고, 플랫폼이 건강한지 살피고, 문제를 추적합니다. 사용자 기획서의 페르소나 **하루**가 여기에 해당합니다.

- 코드: `Admin` 모델. **테넌트에 속하지 않는 전역 주체**입니다.
- 사용 규칙: 영어로는 **Admin**, 한국어로는 **운영자**. 한 문서 안에서는 하나로 통일합니다.
- 헷갈리기 쉬운 점: 테넌트의 회원도 "관리자(MANAGER)" 역할을 가질 수 있지만, 그건 **운영자(Admin)가 아닙니다.** 운영자는 플랫폼 전체를, 테넌트 관리자는 자기 커뮤니티만 다룹니다.

### Tenant / 사업자
Blommunity를 빌려 자기 커뮤니티를 운영하는 **고객**입니다. 동호회 운영자, 1인 크리에이터, 사내 라운지 담당자 등. 페르소나 **지은**이 여기에 해당합니다.

- 코드: `Tenant` 모델. 모든 테넌트 스코프 데이터의 뿌리입니다.
- 사용 규칙: 영어로는 **Tenant**, 한국어로는 **사업자** 또는 **테넌트**. **두 단어 혼용을 허용**하되, **한 문서 안에서는 하나로 일관**되게 씁니다(기능·아키텍처 문서는 "테넌트", 제품·사용자 기획서는 "사업자"를 선호).
- 헷갈리기 쉬운 점: 테넌트는 "회사/조직 하나"이고, 그 안의 사람(첫 가입자 포함)은 모두 **User**입니다.

### User / 회원 / 사용자
특정 테넌트에 소속된 **최종 이용자**입니다. 글을 쓰고 댓글을 달고 공간·게시판에 참여합니다. 페르소나 **민수**가 여기에 해당합니다.

- 코드: `User` 모델. 항상 `tenantId`를 가집니다.
- 사용 규칙: 영어로는 **User**, 한국어로는 **회원** 또는 **사용자**. 문맥에 따라 둘 다 쓰되 한 문서 안에서는 일관되게.
- 헷갈리기 쉬운 점: 테넌트 회원가입(T-AC-01)으로 만들어지는 최초 인물도 `User`입니다(역할이 `OWNER`인 User). 즉 "사업자=Tenant", "사업자 계정으로 일하는 사람=OWNER 역할의 User"입니다.

## 2. 도메인 계층 용어

| 용어 | 영어 | 코드 모델 | 정의 |
|------|------|-----------|------|
| 공간 | Space | `Space` | 한 테넌트가 운영하는 사이트/커뮤니티 단위. 한 테넌트가 여러 개를 가질 수 있습니다. |
| 게시판 | Board | `Board` | 공간 안의 주제별 게시판. 공간에 종속됩니다. |
| 게시물 | Post | `Post` | 게시판에 작성된 글. |
| 댓글 | Comment | `Comment` | 게시물에 달린 댓글. `parentId`로 대댓글 계층을 표현합니다. |

> 계층 관계: **Tenant ─ Space ─ Board ─ Post ─ Comment**. 멤버십은 `SpaceUser`·`BoardUser` 조인 테이블이 담당합니다. 자세한 관계는 [erd.md](erd.md)를 참고하세요.

## 3. 역할·상태·공개 범위

### Role (OWNER / MANAGER / MEMBER)
권한 역할입니다. 인가 계층은 **`OWNER > MANAGER > MEMBER`** 순서입니다.

- **중요: 같은 `Role` enum이 두 맥락에서 재사용됩니다.**
  - **운영자(Admin) 맥락**: 플랫폼 전체에 대한 권한. 예) `OWNER` 운영자는 다른 운영자 권한을 임명할 수 있습니다.
  - **테넌트 사용자(User / SpaceUser / BoardUser) 맥락**: 특정 테넌트·공간·게시판에 대한 권한. 예) 공간의 `OWNER`는 그 공간만 관리합니다.
- 따라서 문서에서 역할을 언급할 때는 **어느 맥락의 역할인지**를 분명히 합니다(예: "공간 Owner", "운영자 Owner").

| 역할 | 영어 | 한 줄 의미 |
|------|------|-----------|
| 소유자 | OWNER | 최고 권한. 권한 임명·삭제까지 가능 |
| 관리자 | MANAGER | 운영 권한. 승인·삭제 일부 가능, 권한 임명은 불가 |
| 일반사용자 | MEMBER | 기본 역할. 자기 활동만 |

### Visibility (PUBLIC / PRIVATE)
공개 범위입니다. `Space`·`Board`·`User`에 적용됩니다.

- **PUBLIC(공개)**: 누구나 접근/조회 가능.
- **PRIVATE(비공개)**: 멤버만. 초대받아야 들어갈 수 있습니다.
- 기본값: Space·Board는 `PUBLIC`, User는 `PRIVATE`.

### MembershipStatus (PENDING / ACTIVE)
멤버십 상태입니다. `SpaceUser`·`BoardUser`에 적용됩니다.

- **PENDING(초대 대기)**: 초대를 받았으나 아직 수락하지 않은 상태.
- **ACTIVE(멤버)**: 실제 멤버. 공개 공간에 직접 가입하거나 초대를 수락하면 됩니다.
- 상태 전이: `(없음) → PENDING → ACTIVE → (삭제)`. 거절/회수/탈퇴는 레코드를 실제로 삭제합니다.

## 4. 아키텍처·일반 용어

### Headless / 헤드리스
화면(프론트엔드)과 데이터·로직(백엔드)을 분리해, 백엔드가 **API로만** 기능을 제공하는 방식입니다. Blommunity v1.0의 정체성은 **"헤드리스 백엔드 1.0"** — API가 안정되어 신뢰하고 의존해도 된다는 약속입니다. 화면(콘솔/어드민)은 v2.0 이후로 미룹니다. 자세한 의미는 [roadmap.md](roadmap.md)를 참고하세요.

### MACH
현대적 백엔드 아키텍처 지향을 가리키는 약어입니다.
- **M**icroservices(마이크로서비스 지향): 도메인별 독립 배포 가능하게 설계
- **A**PI-first(API 우선): 화면보다 API를 먼저·중심으로
- **C**loud-native(클라우드 친화): 클라우드 환경에서 잘 도는 구조
- **H**eadless(헤드리스): 위 정의 참고

### 멀티테넌트 / Multi-tenant
하나의 시스템·DB로 **여러 테넌트(사업자)**를 동시에 서비스하되, 각 테넌트의 데이터는 서로 보이지 않게 격리하는 방식입니다. Blommunity는 테넌트 스코프 모델마다 `tenant_id`를 두어 격리합니다. 격리를 코드 누락 없이 보장하는 전략은 [ADR 0001](adr/0001-tenant-isolation.md)에서 다룹니다.

### 테넌트 격리 / Tenant Isolation
한 테넌트의 데이터가 다른 테넌트에게 절대 노출되지 않도록 막는 것. Blommunity에서 **가장 중요한 보안 속성**입니다(X-SC-07). 현재는 쿼리마다 `tenant_id` 필터를 수동 전달하는데, 단 한 곳이라도 빠지면 격리가 깨지므로 자동 강제가 필요합니다.

### 소프트 삭제 / Soft Delete
레코드를 실제로 지우지 않고 `deletedAt`에 시각만 찍어 "삭제된 것으로 취급"하는 방식(X-DT-01). 복구가 가능해 운영 안정성이 높습니다. 단, 멤버십 조인 테이블과 감사 로그는 예외입니다([erd.md](erd.md) 참고).
