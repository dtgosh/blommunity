# Blommunity API 버저닝 정책

> v1.0의 정체성은 **"헤드리스 백엔드 1.0 — API가 굳었고, 신뢰하고 의존해도 된다"**입니다. 그 약속을 지키는 규칙이 이 문서입니다.
>
> 사업자는 자기 프론트엔드를 Blommunity API 위에 올립니다. API가 예고 없이 바뀌면 그들의 서비스가 깨집니다. 이 문서는 **무엇이 깨지는 변경인지, 어떻게 예고하는지**를 못 박아, 의존해도 안전하게 만듭니다.

---

## 1. 버전 표기 — URL prefix

API 버전은 **URL 경로 prefix**로 표기합니다.

```
/v1/posts
/v1/spaces/{id}
/v2/...   (먼 미래)
```

- **결정**: 헤더 기반(`Accept: application/vnd.blommunity.v1+json`)이나 쿼리 파라미터 방식이 아니라 **URL prefix(`/v1/`)**를 씁니다.
- **이유**: 가장 눈에 보이고, 캐시·로그·디버깅에서 버전이 즉시 드러나며, 사업자가 어느 버전을 쓰는지 헷갈릴 여지가 적습니다. 헤드리스 API의 단순·명료 원칙에 맞습니다.
- 모든 공개 엔드포인트는 버전 prefix를 가집니다. 버전 없는 경로(`/health` 등 운영 엔드포인트)는 버전 정책 대상이 아닙니다.

---

## 2. Breaking Change(깨지는 변경)의 정의

다음은 **메이저 버전을 올려야 하는** 변경입니다. 같은 메이저 버전(`/v1/`) 안에서는 **하지 않습니다.**

- 응답에서 **필드 제거** 또는 필드의 **의미·타입 변경**(예: string → number, 날짜 포맷 변경)
- **응답 형식(구조) 변경** — 객체를 배열로, 중첩 구조 변경, 래핑 형태 변경 등
- 요청에 **필수 파라미터 추가** 또는 기존 파라미터를 필수로 승격
- 엔드포인트 **경로·HTTP 메서드 변경**, 엔드포인트 **제거**
- **에러 응답 형식**의 구조 변경, 기존 에러 코드의 의미 변경
- 기존 동작의 **기본값 변경**으로 결과가 달라지는 경우
- 인증·인가 요구사항을 더 엄격하게 바꾸는 변경(기존 호출이 거부되게 되는 경우)

> 판단 기준 한 줄: **"기존 클라이언트가 코드를 안 고치면 깨지는가?"** 깨지면 breaking입니다.

---

## 3. 마이너 버전에서 허용되는 변경(비파괴)

다음은 같은 메이저(`/v1/`) 안에서 **자유롭게** 할 수 있습니다. 클라이언트는 이런 변경에 견디도록 만들어야 합니다.

- 응답에 **새 필드 추가**(클라이언트는 모르는 필드를 무시해야 함)
- **새 엔드포인트** 추가
- **선택(optional) 요청 파라미터** 추가
- 새로운 **선택적** 정렬·필터 옵션 추가
- 버그 수정으로 **문서화된 동작에 맞게** 바로잡는 변경
- 성능 개선, 에러 메시지 문구 개선(구조 불변)

> 그래서 클라이언트 구현 규칙: **모르는 필드는 무시하고, 필요한 필드만 읽는다.** 이 관용성(tolerant reader)이 깨지지 않는 통합의 핵심입니다.

---

## 4. Deprecation(폐기 예고) 정책

기능을 없애거나 다음 메이저로 옮길 때는 **조용히 끊지 않고 예고**합니다.

- **예고 기간**: 폐기 공지 후 **최소 6개월**간 기존 동작을 유지합니다(메이저 전환 시). 작은 비파괴 정리는 더 짧을 수 있으나, 깨지는 변경은 반드시 메이저 + 예고 기간을 거칩니다.
- **HTTP 헤더로 알립니다.**
  - `Deprecation: true` (또는 폐기 시점 날짜)
  - `Sunset: <RFC 1123 날짜>` — 해당 시점 이후 동작이 사라질 수 있음
  - `Link: <문서 URL>; rel="deprecation"` — 마이그레이션 안내로 연결
- **문서 표시**: 폐기 대상 엔드포인트·필드는 API 문서(Swagger, X-OP-02)에 `deprecated` 표시와 대체 방법을 명시합니다.
- **변경 로그**: 모든 폐기·신규·변경은 CHANGELOG와 릴리스 노트에 기록합니다(`commit-and-tag-version` 흐름과 연계).

## 5. Semantic Versioning과의 매핑

제품 버전(SemVer `MAJOR.MINOR.PATCH`)과 API 버전(`/vN/`)의 관계입니다.

| 제품 버전 변화 | 의미 | API 영향 |
|----------------|------|----------|
| **PATCH** (0.1.1 → 0.1.2) | 버그 수정 | API 표면 불변 |
| **MINOR** (0.1.0 → 0.2.0) | 기능 추가(하위 호환) | `/v1/` 유지, 필드·엔드포인트 추가 가능 |
| **MAJOR** (1.x → 2.0) | 호환을 깨는 변경 | 새 `/v2/` 도입, `/v1/`은 예고 기간 동안 병행 |

- **API 메이저 버전(`/vN/`)은 제품 MAJOR와 함께** 올라갑니다. 즉 `/v2/`는 제품 2.0과 함께 등장합니다.
- v1.0~v1.x 동안 `/v1/`은 **안정 표면**입니다. 이 구간에서 깨지는 변경은 없습니다 — 그것이 "v1.0에서 API가 굳었다"는 약속의 구체적 내용입니다.
- 로드맵상 v1.0까지는 `/v1/` 하나만 존재합니다. `/v2/`는 콘솔·어드민 UI와 함께 오는 먼 단계입니다([roadmap.md](roadmap.md) 참고).

> v0.x 동안에는 빠른 발전을 위해 `/v1/` 내부에서도 변경이 있을 수 있으나, **v1.0.0 태깅 순간부터** 위 규칙이 엄격히 적용됩니다. v1.0은 "이제부터 의존해도 된다"는 분기점입니다.
