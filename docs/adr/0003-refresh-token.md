# ADR 0003 — Refresh Token 도입

- **Status**: Accepted
- **작성일**: 2026-05-29
- **기준**: blommunity v0.1.1
- **관련**: X-SC-08(RefreshToken 회전), T-AC-06(토큰 갱신), [threat-model.md](../threat-model.md) S 범주

> 인증 토큰의 수명과 갱신 방식을 결정합니다.

---

## Context

현재 인증은 **access token만** 발급합니다(`apps/api/src/auth/auth.controller.ts`의 sign-in/sign-up이 JWT 문자열 하나를 반환). 그리고 `auth.module.ts`에는 **만료 시간(`expiresIn`)이 설정되어 있지 않습니다.** 즉 사실상 장수명 토큰 하나로 인증이 유지됩니다.

이 구조는 양쪽으로 곤란합니다.

- **토큰을 짧게 만들면 UX가 나빠집니다.** 예를 들어 access token을 15분으로 잡으면, 사용자는 15분마다 다시 로그인해야 합니다. 커뮤니티 서비스에서 이건 받아들이기 어렵습니다.
- **토큰을 길게(또는 무기한으로) 두면 탈취 위험이 큽니다.** 한 번 새어 나간 토큰이 오래(혹은 영원히) 유효하면, 공격자가 그 창 동안 자유롭게 행동합니다. [위협 모델](../threat-model.md)의 Spoofing 범주에서 가장 큰 보강 포인트로 지목한 부분입니다.

"짧게 해서 안전 + 길게 느껴져서 편함"을 동시에 얻으려면 토큰을 **두 종류로 나눠야** 합니다.

---

## Decision

**Access Token(15분) + Refresh Token(14일)** 구조로 분리하고, Refresh Token에 **회전(rotation)**을 적용합니다.

- **Access Token (AT) — 15분.** 실제 API 호출에 쓰는 단명 토큰. 짧으므로 탈취돼도 유효 창이 좁습니다.
- **Refresh Token (RT) — 14일.** AT를 재발급받기 위한 장명 토큰. API 호출에는 쓰지 않고 **재발급 엔드포인트에서만** 씁니다.
- **회전(rotation)**: RT를 사용해 새 AT를 받을 때, **RT도 새로 발급하고 직전 RT는 무효화**합니다. 한 RT는 한 번만 쓰입니다.
  - 추가 안전장치(권장): 이미 쓰인(무효화된) RT가 다시 들어오면 **탈취 의심**으로 보고 해당 사용자의 RT 패밀리를 전부 무효화합니다(재사용 감지).

흐름:

```
[로그인] → AT(15분) + RT(14일) 발급
   ↓ (15분 경과, AT 만료)
[API 호출 → 401]
   ↓
[RT로 재발급 요청] → 새 AT + 새 RT 발급, 이전 RT 무효화 (회전)
   ↓
[API 호출 재개]
```

---

## Consequences

### 좋아지는 점
- **탈취 토큰의 유효 창이 좁아집니다.** AT가 15분이라, 새어도 곧 만료됩니다.
- **재사용 감지가 가능합니다.** 회전 덕분에 "한 번 쓴 RT가 또 들어오는" 비정상을 탐지해 세션을 끊을 수 있습니다.
- **UX가 유지됩니다.** 클라이언트는 401을 받으면 RT로 조용히 재발급해 사용자가 끊김을 느끼지 않습니다(silent refresh).

### 감수해야 하는 점 / 클라이언트 영향
- **클라이언트가 재발급 로직을 갖춰야 합니다.** API 호출이 401이면 RT로 재발급을 시도하고, 실패하면 로그인 화면으로 보냅니다. 이 흐름은 API 사용 가이드에 명시합니다(T-AC-06).
- **RT 저장소가 필요합니다.** 무효화·회전을 하려면 발급된 RT의 유효 상태를 서버가 알아야 합니다(아래 "저장 위치" 참고).
- 동시 요청이 한꺼번에 401을 받아 동시에 재발급을 시도하는 경합을 클라이언트가 다뤄야 합니다(재발급 1회로 묶기).

---

## Alternatives Considered

### A. Access Token만 길게 — 기각
지금처럼 토큰 하나를 쓰되 수명을 길게 둡니다. 단순합니다.

**기각 이유:** 탈취 시 피해가 큽니다. 무효화 수단도 마땅치 않아(JWT는 기본적으로 stateless) 한 번 새면 만료까지 막을 길이 없습니다.

### B. 세션 쿠키 기반 인증 — 기각
서버 세션 + 쿠키로 인증합니다.

**기각 이유:** Blommunity는 **헤드리스 API**가 정체성입니다. 사업자가 자기 프론트(웹/앱)를 자유롭게 붙이는 구조라, 브라우저 쿠키·CSRF·도메인 종속이 전제되는 세션 방식은 부합하지 않습니다. 토큰 기반(Bearer)이 헤드리스·크로스 플랫폼에 맞습니다.

## 저장 위치 (결정 미룸 — v0.2.0 구현 시 확정)

RT의 유효 상태를 어디에 둘지는 **이 ADR에서 확정하지 않고 비교만** 합니다. 추천은 제시하되, 최종 결정은 v0.2.0 구현 시점에 다시 봅니다.

| 기준 | DB(별도 테이블) | Valkey |
|------|------------------|--------|
| 영속성 | 강함(재시작에도 유지) | 약함(재시작 시 전부 무효) |
| 회전 이력 추적 | 모든 회전 기록 남기기 쉬움 | TTL로 자동 만료, 이력은 별도 |
| 만료 처리 | 직접 정리(배치) 필요 | **TTL 자동 만료** |
| 성능 | 쓰기 부담 약간(매 회전마다 write) | **빠름** |
| 운영 | 마이그레이션 대상 | 이미 스택에 있음(`@app/cache`, Valkey 9) |

- **DB(별도 테이블)**: 영속성이 강하고 모든 회전을 기록으로 추적할 수 있습니다. 대신 매 회전마다 write가 생겨 쓰기 부담이 약간 있고, 만료된 RT를 주기적으로 정리해야 합니다.
- **Valkey**: 빠르고 TTL로 만료가 자동입니다. 다만 재시작 시 모든 RT가 무효화됩니다 — 이건 양면입니다. 사용자는 다시 로그인해야 해서 불편할 수 있지만, "비상시 전체 세션을 한 번에 끊는" 수단이 되기도 합니다.

> **추천: Valkey.** 성능과 TTL 자동화의 이점이 크고, **이미 스택에 Valkey 9가 있습니다**(`compose.dev-infra.yaml`, `@app/cache`, X-OP-05). RT는 본질적으로 만료가 있는 단기 자격이라 캐시 저장소와 성격이 맞습니다. 다만 "재시작 시 전체 무효" 트레이드오프와 재사용 감지(패밀리 무효화)를 어떻게 구현할지는 **v0.2.0 구현 시 재확인**합니다. 회전 이력의 감사 추적이 중요해지면 DB 병행도 검토합니다.
